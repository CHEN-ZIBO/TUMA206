# Water Treatment Plant Digital Twin

**TUMA206 — Modern Developments in Industry | TUM Asia | 2025/26 Semester 2**

A fully-local, runnable digital twin of a water treatment process. Built in Python
using MQTT for messaging, Streamlit for the operator dashboard, and CSV for
time-series logging. The system demonstrates the complete industrial stack:
field sensors → simulated PLC → messaging layer → historian → dashboard → AI assistant.

For the full architecture rationale see [`docs/architecture.md`](docs/architecture.md).

## Architecture Overview

```
┌──────────────────────────────────────────────────────────────────┐
│  Level 3/4  AI Assistant + Operator Dashboard  (dashboard.py)    │
│             Streamlit web UI · localhost:8501                     │
├──────────────────────────────────────────────────────────────────┤
│  Level 2    Historian  (historian.py)                            │
│             data/historian.csv · 3600-row in-memory ring buffer  │
├──────────────────────────────────────────────────────────────────┤
│  Level 2    MQTT Broker  (Mosquitto · localhost:1883)            │
│             Topics: /tags  /alarms  /faults  /commands           │
├──────────────────────────────────────────────────────────────────┤
│  Level 1    Simulated PLC  (plc_controller.py)                   │
│             1-second scan cycle · state machine · on/off + PID   │
├──────────────────────────────────────────────────────────────────┤
│  Level 0    Sensor Simulator  (sensor_simulator.py)              │
│             noise / stuck / drift / spike fault modes            │
├──────────────────────────────────────────────────────────────────┤
│  Level 0    Process Simulator  (process_simulator.py)            │
│             T-101 tank · P-101 pump · V-101 valve · DP-101 dose  │
└──────────────────────────────────────────────────────────────────┘
```

## Tech Stack

| Component | Choice | Reason |
|-----------|--------|--------|
| Language | Python 3.10+ | Simulation, control logic, data handling and AI in one language |
| MQTT client | paho-mqtt | Standard IIoT library; pub/sub mirrors industrial field-to-SCADA pattern |
| MQTT broker | Mosquitto | Lightweight, zero-config, stable for local demo |
| Dashboard | Streamlit | Web UI with no frontend code; real-time widgets built-in |
| Historian | CSV + deque ring buffer | Zero extra dependencies; human-readable; proves concept at viva |
| AI assistant | Rule-based + optional LLM | Works fully offline; LLM is an opt-in upgrade |

## Running the Full System

Three terminals, started in this order:

```bash
# Terminal 1 — MQTT broker
mosquitto -v

# Terminal 2 — main simulator loop (process + PLC + MQTT + historian)
python src/main.py

# Terminal 3 — operator dashboard
streamlit run src/dashboard.py
```

Open **http://localhost:8501** in your browser.

#### Fault mode smoke test

Open a Python REPL from the repo root:

```python
import sys; sys.path.insert(0, 'src')
from process_simulator import ProcessSimulator
from sensor_simulator  import SensorSimulator
import config

p = ProcessSimulator()
s = SensorSimulator(p)
p.apply_commands(pump_on=True, valve_open=False, dosing_on=False)

# normal reading
p.step(1); print(s.read_all_tags()['WT_A101_PH_PV'])

# stuck fault — pH freezes
s.set_fault('WT_A101_PH_PV', 'stuck')
for _ in range(5):
    p.step(1)
    print('sensor:', s.read_tag('WT_A101_PH_PV'),
          '  true:', round(p.state['ph'], 3))
```

Expected: sensor value stays constant while true value drifts.

#### Pass criteria

#### Integration smoke test

```python
import sys; sys.path.insert(0, 'src')
import config
from process_simulator import ProcessSimulator
from sensor_simulator  import SensorSimulator
from plc_controller    import PLCController

config.TANK_CAPACITY_L = 100   # small tank so transitions happen in ~100 steps
p = ProcessSimulator(); p.level_pct = 25.0
s = SensorSimulator(p)
plc = PLCController()
p.apply_commands(False, False, False)

for i in range(120):
    p.step(config.SAMPLE_RATE_S)
    tags = s.read_all_tags()
    cmds = plc.execute(tags)
    p.apply_commands(cmds['WT_P101_PUMP_CMD'],
                     cmds['WT_V101_OUTLET_CMD'],
                     cmds['WT_D101_DOSING_CMD'])
    if i % 20 == 0:
        print(f"step {i:3d} | {cmds['state']:12s} | "
              f"level={tags['WT_T101_LEVEL_PV']:.1f}%")
```

Expected: state transitions from FILLING → TREATING as level rises above 40 %.

#### Alarm smoke test

```python

print(cmds['state'])               # FAULT
```

#### Infra alarm smoke test

```python
import sys, time; sys.path.insert(0, 'src')
import mqtt_client as m

# Patch timeout to 0.5 s for a fast test
m.INFRA_FAULT_TIMEOUT_S = 0.5

client = m.MQTTClient()   # broker unreachable → connect will fail
# Simulate a prior connection then disconnect
import time as _t
with client._lock:
    client._disconnect_since  = _t.monotonic()
    client._last_connected_ts = _t.monotonic() - 1

time.sleep(0.7)
print("infra_alarm:", client.infra_alarm)   # → True
client.disconnect()
```

## Limitations and Future Improvements

- No real PLC hardware — Python simulates the deterministic scan cycle
- pH dynamics are a simplified first-order model, not a real chemical simulation
- CSV historian is sufficient for demo; production would use InfluxDB or TimescaleDB
- AI assistant uses rule-based fallback; LLM API call is an optional upgrade
- No MQTT authentication — acceptable for local demo, not for production

These constraints are intentional for simplicity and classroom reproducibility.
The industrial architecture pattern (ISA-95 Purdue layers, MQTT pub/sub,
historian, dashboard, AI assistant) is preserved in full.
